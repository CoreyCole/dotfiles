# Research Questions: Q-Review Skill

## Brainstorm Summary
- The desired outcome is a new QRSPI `/q-review` skill that begins after implement completion, produces a durable review artifact under `thoughts/.../plans/.../reviews/`, and does not edit code.
- The engineer wants `q-review` to combine two responsibilities in one stage: validate implementation against `plan.md` and review the resulting code for adherence to codebase best practices.
- The review artifact should contain severity-tagged issues plus suggested ways to address them, then stop so the engineer can decide with the agent what to do next.
- Fast verification is in scope for `q-review`; expensive manual or integration testing is intentionally deferred to a later `q-test` stage.
- Important review concerns already called out include meaningful test value vs box-checking, reducing unnecessary mocking, robust error handling, and avoiding swallowed errors.
- A key open tension for research is how `q-review` should fit QRSPI’s current artifact-loading and scout-only reconnaissance rules while also supporting multiple review lanes or subagent perspectives.

## Context
QRSPI now hands completed implementation to `/q-review`, but the skill itself does not exist yet. The repo already contains QRSPI stage skills, a general review extension, a shared review rubric, and a validate-plan draft, so the next step is to research how those existing patterns should constrain a new review-stage skill.

## Questions
1. What exact artifact-loading rules, response shapes, metadata conventions, and handoff expectations do the existing QRSPI skills establish that a new `q-review` skill must follow to fit the pipeline cleanly?
2. At the point where `q-review` starts from an implement-complete handoff, what inputs and evidence are already available in the plan directory, handoff file, codebase, and git history, and how do existing skills or agents consume comparable inputs today?
3. How do the existing review surfaces in this repo — the `/review` extension, `review-rubric`, reviewer agent, and `cc-validate-plan` draft — structure severity, findings, verification, non-editing behavior, and suggested next actions?
4. Given the current QRSPI plan-directory scaffolding and load rules, where should `q-review` write its primary artifact and any supporting context so later stages can find them without breaking existing conventions?
5. What fast automated verification commands and review-time checks are already used or implied in this repo, and how are they separated from slower manual or integration testing that should remain out of scope for `q-review`?
6. What existing mechanisms, conventions, or constraints in this repo would shape a multi-lane review process for concerns like plan adherence, test meaningfulness, error handling, security, and codebase best practices?
