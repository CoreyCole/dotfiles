# Code Review

## Summary
Resume flow wiring is mostly correct: resume route is authenticated, thread ownership is enforced via `GetAgentThreadForUser`, and the active-run guard is enforced at DB level and surfaced as HTTP 409. However, the new partial unique index can hard-fail DB startup on existing data, which is a commit blocker.

## Findings

### [P1] Startup migration can fail on existing duplicate running rows
**File:** `db/migrations/schema.sql:289`
**Issue:** The new unique partial index is added directly:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_agent_runs_thread_running
    ON agent_runs(thread_id)
    WHERE status = 'running';
```

If the existing DB already contains >1 `running` row for the same `thread_id` (possible before this guard existed), index creation fails with `UNIQUE constraint failed: agent_runs.thread_id`. This repository executes the full `schema.sql` at startup (`server/services/db/db.go`), so startup/migration fails hard.

**Impact:** Deploying this change can brick service startup for users with pre-existing duplicate running runs.

**Suggested Fix:** Add a deterministic data cleanup step before creating the index (e.g., keep newest running row per thread, mark older running rows failed), then create the index.
