# Research: Observable software factory direction for agents-chat

## Brainstorm Summary
- The desired outcome is an observable software factory inside `cn-agents/server`, centered on `agents-chat`, where long-running or stuck work is visible in Temporal and later in the cn-agents web UI.
- Existing cn-agents infrastructure should be favored: SQLite is the preferred local datastore, terminal sessions should be web-observable, and `agents-chat` is the main product surface rather than the older experimental `pipelines` UI.
- Linear is the human-visible work system: top-level tickets and nested child tickets should represent durable task structure, and human approvals live in Linear even if cn-agents also shows or advances internal stage state.
- Gas City is a source of ideas, not an assumed dependency. Push-based orchestration is preferred; if polling remains, it should be deterministic Go code and must not wake LLM sessions just to poll.
- Open tensions intentionally deferred to research include how much of the existing `pipelines` / QRSPI Temporal machinery should be reused, how terminal-first then web-pickup flows should map onto workflow state, and which Gas City concepts are still worth preserving once beads and Dolt are removed.

## Detailed Findings

### 1. Existing Temporal workflows, task queues, workflow IDs, UI entry points, and the gap between pipelines and `agents-chat`
- The shared Temporal adapter defines two task queues — `pipeline-go` and `pipeline-ts` — plus three review signals: `continue`, `approve`, and `reject` (`pkg/agents/temporal/manager.go:13-23`). Pipeline stage workflow IDs are deterministic: `WorkflowIDForStage` returns `pipeline-<pipelineID>-<stageID>` (`pkg/agents/temporal/manager.go:123-124`).
- `main.go` conditionally enables Temporal when `CN_TEMPORAL` is true, requires `PIPELINE_ARTIFACTS_DIR` and `PIPELINE_AGENT_NAME`, creates the Temporal manager, starts the Go worker, registers all registry workflows, and separately registers the conversation workflow used by `agents-chat` (`main.go:274-300`). The same file also mounts the observable UI entry points: `/temporal`, `/pipelines`, `/api/pipelines`, `/agent-chat`, `/internal/events`, and `/internal/agent-chat/events` (`main.go:382-391,451-480`).
- QRSPI is already a multi-stage Temporal workflow. Its registry entry declares agent stages (`question`, `research`, `design`, `outline`, `plan`) interleaved with review stages (`review_question`, `review_research`, `review_design`, `review_outline`, `review_plan`) (`pkg/agents/workflows/qrspi/workflow.go:39-48`). Review workflows move the pipeline to `review`, open the three signal channels, call `ContinuePiAgent` when the human sends a message, advance to the next stage on approve, and rerun the reviewed stage on reject (`pkg/agents/workflows/qrspi/workflow.go:169-251`).
- Pipeline creation already starts the first workflow immediately. `CreatePipeline` computes a plan directory under `PIPELINE_ARTIFACTS_DIR/<agent>/<workflow-base-dir>/...`, inserts the pipeline row, builds `registry.StageInput`, computes the deterministic workflow ID, starts the first Temporal workflow, and stores the Temporal run ID (`server/services/pipelines/service.go:87-166`).
- The pipeline UI is already stage/session oriented. The detail page starts review workflows for pending review stages (`server/services/pipelines/handler.go:338-357`), the review panel posts to `/continue`, `/approve`, and `/reject` (`server/services/pipelines/templates.templ:273-307`), and the stage page opens SSE streams for stage content, session logs, and event history (`server/services/pipelines/templates.templ:343-390`, `server/services/pipelines/handler.go:148-239`, `server/services/pipelines/api_handler.go:24-120`).
- `agents-chat` uses the same Temporal infrastructure but not the pipeline state machine. Each run gets a UUID-backed workflow ID `agent-chat-run-<uuid>` (`server/services/agentchat/service.go:303-317`), starts `conversationworkflow.RunTurnWorkflow` with a callback endpoint at `/internal/agent-chat/events` (`server/services/agentchat/service.go:391-430`, `pkg/agents/workflows/conversation/workflow.go:12-19`), and handles events directly as transcript/checkpoint/run-complete/run-failed messages (`server/services/agentchat/handler.go:242-278`).
- The current factual gap is therefore structural, not infrastructural: the pipeline path already has multi-step stage orchestration, deterministic stage IDs, artifact/session records, and review controls, while `agents-chat` currently executes one conversation turn per run and exposes send/resume/fork plus transcript/artifact panes rather than stage progression or approval controls (`server/services/pipelines/templates.templ:238-390`, `server/services/agentchat/page_chat.templ:16-57`, `server/services/agentchat/templates.templ:33-534`).

### 2. How `agents-chat` persists and presents thread/run/entry state, and what already supports terminal-first execution, later web pickup, and follow-up work
- The SQLite model is explicit. `agent_threads` stores `cwd`, `lineage_id`, `head_entry_id`, `parent_thread_id`, and `forked_from_entry_id`; `agent_entries` stores the lineage graph plus `payload_json`, `origin_thread_id`, `origin_run_id`, and `session_timestamp`; `agent_runs` stores `trigger`, `restore_head_entry_id`, `result_head_entry_id`, `workflow_id`, `temporal_run_id`, and `artifact_root` (`db/migrations/schema.sql:231-311`).
- New, resumed, and forked work all create new `agent_runs` rows. `StartThread` creates a new thread and a `send` run (`server/services/agentchat/service.go:99-154`); `ResumeThread` creates a `resume` run using the thread’s current head entry as `restore_head_entry_id` (`server/services/agentchat/service.go:159-197`); and `ForkThread` creates a new thread in the same lineage, records `parent_thread_id` and `forked_from_entry_id`, computes a restore head from the source entry, and creates a `fork` run (`server/services/agentchat/service.go:242-388`).
- The active-run guard is enforced in the database and surfaced in the service/handler. `agent_runs` has a unique partial index on `thread_id` where `status = 'running'` (`db/migrations/schema.sql:309-311`), `createRun` inserts new runs with `status = 'running'` and returns `ErrThreadRunInProgress` on unique-constraint failure (`server/services/agentchat/service.go:303-328`), and `ResumeThread` maps that error to HTTP 409 (`server/services/agentchat/handler.go:107-130`).
- Terminal-first execution is implemented through snapshot replay into a Pi session file. `BuildSnapshot` reconstructs the ancestry path for the selected head entry using the recursive `ListAgentEntryPath` query (`server/services/agentchat/service.go:504-548`, `db/queries/agent_entries.sql:31-80`). The TS worker then writes a `session.jsonl` with the saved header plus all prior `payload_json` entries and reopens it through `SessionManager.open(...)` (`pkg/agents/temporal/workers/ts/conversation.ts:20-40`).
- Checkpointing is incremental. The TS worker diffs the current session against the incoming snapshot, emits only new entries as a `Checkpoint`, and includes the current leaf as `head_entry_id` (`pkg/agents/temporal/workers/ts/conversation.ts:43-79`). The Go side persists those entries, updates the thread/run head pointers, commits, then patches the stable transcript, artifact pane, and sidebar (`server/services/agentchat/service.go:553-614`).
- Later web pickup is already supported by reconstruction rather than by replaying a stored session pointer. `BuildPageArgs` loads the latest thread/run, rebuilds the stable transcript from persisted entries, rebuilds live transcript state from the in-memory reducer, and rebuilds the artifact pane from the filesystem under the thread cwd (`server/services/agentchat/service.go:675-735`). `BuildArtifactPane` scans renderable files under the cwd, can refocus onto a plan root, and renders the selected file (`server/services/agentchat/service.go:738-862,1211-1269`).
- One important limitation is that `ConversationRunResult` includes `session_path`, but `FinalizeRun` does not persist it to `agent_runs`; it only marks the run complete and stores `result_head_entry_id` (`pkg/agents/conversation/types.go:70-76`, `server/services/agentchat/service.go:618-633`). So later pickup in `agents-chat` is driven by persisted entries plus filesystem scanning and separate Pi session discovery, not by a run-to-session foreign key.
- Follow-up context is already encoded in the model. `lineage_id`, `head_entry_id`, `parent_thread_id`, `forked_from_entry_id`, `parent_entry_id`, `origin_run_id`, and `restore_head_entry_id` preserve where a resumed or forked conversation starts and how it branches (`db/migrations/schema.sql:231-311`, `server/services/agentchat/service.go:331-388`). The fork UI submits `source_thread_id` and `source_entry_id` directly from transcript messages (`server/services/agentchat/templates.templ:343-366`).
- `agents-chat` also already knows how to surface plan-related context from outside its own tables. It scans `~/.pi/agent/sessions` for `.jsonl` files, extracts `plan-classification` custom entries when present, falls back to `header.Cwd`, normalizes any path under `thoughts/.../plans/...` to a plan root, and merges those Pi sessions into the sidebar with a `Pi` badge (`server/services/agentchat/service.go:866-1027,1211-1223`, `server/services/agentchat/args.go:26-55`, `server/services/agentchat/templates.templ:33-91`).

### 3. Overlap and divergence between `agents-chat` and the experimental `pipelines` service
- The overlap is mostly infrastructural. Both services are constructed with the same Temporal manager in `main.go` (`main.go:308-322`), both rely on the same two Temporal task queues (`pkg/agents/temporal/manager.go:13-24`), and both use SSE/notifier-driven web updates (`server/services/pipelines/handler.go:27-46,148-239`, `server/services/agentchat/handler.go:24-34,53-82`).
- Both paths are also artifact-root aware, but they persist artifact/session state differently. Pipelines persist stage artifacts in `pipeline_artifacts`, including `session_path` (`db/migrations/schema.sql:217-224`, `pkg/agents/pipeline/activities.go:104-160`). `agents-chat` persists thread/run/entry state in `agent_threads`, `agent_entries`, and `agent_runs`, but not `session_path` (`db/migrations/schema.sql:231-311`, `server/services/agentchat/service.go:618-633`).
- Pipelines are stage-based and deterministic. A pipeline row carries `workflow_type`, `stage`, `status`, `ticket_id`, `base_dir`, and `temporal_run_id` (`db/migrations/schema.sql:195-215`, `db/queries/pipelines.sql:1-29`); stage workflows are selected through the registry (`pkg/agents/registry/registry.go:31-162`); workflow IDs are derived from `(pipelineID, stageID)` (`pkg/agents/temporal/manager.go:123-124`); and review state is advanced by Temporal signals (`server/services/pipelines/service.go:178-296`).
- `agents-chat` is thread/run based rather than stage based. It generates UUID-backed workflow IDs (`server/services/agentchat/service.go:303-317`), starts a single conversation workflow per run (`server/services/agentchat/service.go:391-430`), and advances state by appending entries and updating the thread head (`server/services/agentchat/service.go:553-633`).
- The review flow materially diverges. Pipeline review is explicit, stage-scoped, and signal-driven (`pkg/agents/workflows/qrspi/workflow.go:169-251`, `server/services/pipelines/templates.templ:273-307`). `agents-chat` has no concept of `approve` / `reject` / `continue`; its interaction model is send/resume/fork (`server/services/agentchat/page_chat.templ:42-57`, `server/services/agentchat/templates.templ:343-366`).
- Artifact/session handling also diverges. Pipeline stage pages show artifact trees, session lists, session logs, and event panes backed by `pipeline_artifacts` and session-log endpoints (`server/services/pipelines/service.go:455-525`, `server/services/pipelines/templates.templ:354-390`, `server/services/pipelines/session_logs.go:46-278`). `agents-chat` builds its artifact tree by walking the current cwd or plan root and separately discovers plan-scoped Pi sessions (`server/services/agentchat/service.go:738-862,866-1027,1211-1269`).
- One concrete divergence in session continuation is that the pipeline stage page lets the user switch an “active session”, but `ResumeSessionWithPrompt` only validates that `session_path` is non-empty and then calls `ContinuePipeline`; the actual review workflow continues using `previousArtifact.SessionPath` from the reviewed stage, not the posted `session_path` (`server/services/pipelines/service.go:222-245`, `pkg/agents/workflows/qrspi/workflow.go:176-194`).

### 4. Current Gas City usage that depends on beads, Dolt, patrol/reconciler polling, or order execution, and which concepts are already structurally separate from those loops
- The patrol/reconciler layer is explicit. The city config sets a daemon patrol interval of 30 seconds (`.gascity/city.toml:31-38`), and the supervisor pack defines always-on `deacon` and `witness` sessions (`.gascity/packs/actual/supervisor/pack.toml:15-42`).
- The `deacon` formula depends on beads and order-style routing. It repeatedly queries open, unassigned beads by label, slings them to a specific agent, assigns them to avoid redispatch, runs `gc status`, and then re-pours itself with `bd mol pour mol-deacon-patrol` (`.gascity/packs/actual/supervisor/formulas/mol-deacon-patrol.formula.toml:1-119`).
- The `witness` formula depends on beads plus session state. It lists `in_progress` beads, compares them to `gc session list --json`, reopens orphaned beads, reports queue depth by label, and re-pours itself (`.gascity/packs/actual/supervisor/formulas/mol-witness-patrol.formula.toml:1-54`).
- Deterministic checks also exist as order files. `beads-health.toml` runs `gc beads health --quiet` every 30 seconds under a cooldown gate (`.gascity/orders/beads-health.toml:1-6`), and planner intake is a condition-gated shell check over `gc bd --rig cn-agents list --label=needs-plan ...` (`.gascity/packs/actual/planner/orders/planner-intake.toml:1-6`).
- Planner pack config also depends on bead discovery. It uses `work_query = "gc bd --rig cn-agents list --label=needs-plan --status=open --json ..."` with `mode = "on_demand"` (`.gascity/packs/actual/planner/pack.toml:28-45`).
- Hook-based coordination is also part of the runtime contract. The Pi hook extension injects `gc prime --hook`, `gc nudge drain --inject`, and `gc mail check --inject` before agent start, then runs `gc hook --inject` on shutdown (`.gascity/.pi/extensions/gc-hooks.js:25-46`).
- The source does distinguish one-shot handoff primitives from patrol loops. Planner’s prompt treats routed work as coming either from `gc sling` or `bd ready --metadata-field gc.routed_to=... --unassigned` (`.gascity/packs/actual/planner/prompts/planner.md.tmpl:22,62`). Its handoff section creates downstream beads with `gc.routed_to` metadata, then sends direct notifications with `gc mail send architect ...` / `gc mail send designer ...` (`.gascity/packs/actual/planner/prompts/planner.md.tmpl:107-124`). The prompt also states, verbatim, “The label triggers each agent’s intake order gate” (`.gascity/packs/actual/planner/prompts/planner.md.tmpl:140-144`).
- That means the live code already separates two categories of behavior:
  1. patrol/order/hook loops that keep scanning and recovering work (`.gascity/city.toml:31-38`, `.gascity/packs/actual/supervisor/formulas/mol-deacon-patrol.formula.toml:1-119`, `.gascity/packs/actual/supervisor/formulas/mol-witness-patrol.formula.toml:1-54`, `.gascity/orders/beads-health.toml:1-6`, `.gascity/.pi/extensions/gc-hooks.js:25-46`)
  2. direct handoff/routing contracts built from `gc sling`, `gc mail`, labels, and `gc.routed_to` metadata (`.gascity/packs/actual/planner/prompts/planner.md.tmpl:22,62,107-124,140-144`, `.gascity/packs/actual/supervisor/formulas/mol-deacon-patrol.formula.toml:7-98`)
- I could not find any live source dependency on Dolt beyond `.gitignore` ignoring `.dolt/` and `*.db` as “Beads / Dolt files” (`.gitignore:37-40`). Repository-wide search surfaced Dolt references only in historical thoughts/handoff documents, not in the active server/runtime code.

### 5. Current Linear-related integration surfaces, and the code paths that would participate if Linear became the durable source of truth for task structure and approvals
- The persistent schema is small. `projects` stores `linear_team_id` and `pipelines` stores `ticket_id` (`db/migrations/schema.sql:183-215`). The corresponding query files insert/update those fields but nothing more (`db/queries/projects.sql:1-13`, `db/queries/pipelines.sql:1-29`).
- Pipeline UI/service code already carries `ticket_id` end to end. The create form renders an optional ticket field (`server/services/pipelines/templates.templ:157-214`), the handler binds it (`server/services/pipelines/handler.go:436-445`), the request struct carries it (`server/services/pipelines/args.go:146-152`), and `CreatePipeline` stores it and also copies it into the first `registry.StageInput` (`server/services/pipelines/service.go:87-166`). When review workflows are started later, `ticket_id` is copied back into the review-stage `StageInput` if present (`server/services/pipelines/service.go:247-296`).
- The current approval path does not use Linear. `ApprovePipeline`, `RejectPipeline`, and `ContinuePipeline` all derive the current stage workflow ID and send Temporal signals; none of them call Linear code (`server/services/pipelines/service.go:178-218`). The review UI is therefore Temporal-backed, not Linear-backed (`server/services/pipelines/templates.templ:273-307`).
- Direct Linear issue/project logic lives in the CLI. `cmd/cn-runtime/main.go` registers `tkt` and `proj` subcommands (`cmd/cn-runtime/main.go:16-29`). `tkt` requires `LINEAR_API_KEY` (`cmd/cn-runtime/tkt.go:122-124`), can read an issue including its parent and children (`cmd/cn-runtime/tkt.go:183-203`), create issues and sub-issues via `parentId` (`cmd/cn-runtime/tkt.go:506-706`), add comments (`cmd/cn-runtime/tkt.go:755-809`), change workflow state and optionally add a comment (`cmd/cn-runtime/tkt.go:942-1084`), and replace issue descriptions from a file (`cmd/cn-runtime/tkt.go:1091-1144`). `proj` can query projects by name/slug, create projects, read project details, and post project status updates (`cmd/cn-runtime/proj.go:25-148,170-429`).
- Nested Linear task structure therefore already exists in live code, but only in the CLI path: `issueByIdentifierQuery` returns both `parent` and `children.nodes`, and `tkt create-sub` supplies `parentId` to the mutation (`cmd/cn-runtime/tkt.go:183-203,632-706`).
- If Linear became the durable human-visible source of truth, the participating live code paths are the ones where those identifiers already enter or where approvals currently happen: `db/migrations/schema.sql`, `db/queries/projects.sql`, `db/queries/pipelines.sql`, `server/services/pipelines/args.go`, `server/services/pipelines/service.go`, `server/services/pipelines/handler.go`, and `server/services/pipelines/templates.templ`, plus the existing Linear GraphQL logic in `cmd/cn-runtime/tkt.go` and `cmd/cn-runtime/proj.go`. That is a statement about current code placement, not a design recommendation.
- One notable detail: `tkt finish` says “Mark a ticket as done”, but it actually calls `updateIssueState(..., stateNameInReview, ...)`, where `stateNameInReview = "In Review"` (`cmd/cn-runtime/tkt.go:965-1074`). So the live CLI already distinguishes work-in-progress vs review-state transitions, but it does not currently express a separate approval concept inside the server pipeline flow.
- I could not find a live `server/services/linear` or `pkg/linear` package. Repository-wide search for live GraphQL Linear mutations only surfaced `cmd/cn-runtime`, not server runtime code.

### 6. Current polling loops or deterministic background checks in `.gascity`, what state they inspect or mutate, and what equivalent behaviors currently have no replacement elsewhere in the repo
- The current deterministic background behaviors in `.gascity` are:
  - provider health checking via `gc beads health --quiet` (`.gascity/orders/beads-health.toml:1-6`)
  - label-based routing of open, unassigned beads plus assignee mutation (`.gascity/packs/actual/supervisor/formulas/mol-deacon-patrol.formula.toml:1-119`)
  - orphan recovery by comparing `in_progress` beads to active sessions and resetting stale work to open (`.gascity/packs/actual/supervisor/formulas/mol-witness-patrol.formula.toml:1-54`)
  - queue-depth reporting over label buckets (`.gascity/packs/actual/supervisor/formulas/mol-witness-patrol.formula.toml:29-54`)
  - intake gating over `needs-plan` beads (`.gascity/packs/actual/planner/orders/planner-intake.toml:1-6`, `.gascity/packs/actual/planner/pack.toml:38-45`)
  - hook-time prompt injection for prime/nudge/mail and shutdown-time reconciliation (`.gascity/.pi/extensions/gc-hooks.js:25-46`)
- The state they inspect is also explicit in those files:
  - open or unassigned beads by label (`gc bd --rig ... list --label=... --status=open --no-assignee`) (`.gascity/packs/actual/supervisor/formulas/mol-deacon-patrol.formula.toml:7-98`, `.gascity/packs/actual/planner/orders/planner-intake.toml:1-6`)
  - `in_progress` beads (`bd list --status=in_progress --json`) (`.gascity/packs/actual/supervisor/formulas/mol-witness-patrol.formula.toml:7-16`)
  - active session aliases (`gc session list --json`) (`.gascity/packs/actual/supervisor/formulas/mol-witness-patrol.formula.toml:12-16`)
  - provider health (`gc beads health --quiet`) (`.gascity/orders/beads-health.toml:1-6`)
  - pending routed work visible through hook output, mail, or `gc.routed_to` metadata (`.gascity/.pi/extensions/gc-hooks.js:25-40`, `.gascity/packs/actual/planner/prompts/planner.md.tmpl:22,62,107-124`)
- The state they mutate is also explicit:
  - setting bead assignees during dispatch (`gc bd --rig "$RIG" update <id> --assignee=...`) (`.gascity/packs/actual/supervisor/formulas/mol-deacon-patrol.formula.toml:9-98`)
  - reopening orphaned beads by clearing assignee and setting `status=open` (`.gascity/packs/actual/supervisor/formulas/mol-witness-patrol.formula.toml:20-27`)
  - injecting additional system prompt content before agent start (`.gascity/.pi/extensions/gc-hooks.js:25-40`)
- I could not determine from code alone which of these behaviors an `agents-chat`-first architecture is intended to keep or drop. The live code only shows that these behaviors are presently being provided by `.gascity`, and I did not find an alternate implementation elsewhere in `server/`, `pkg/agents/`, or `cmd/cn-runtime/` that already replaces health checking, orphan recovery, label-based dispatch, or hook-time work injection.
