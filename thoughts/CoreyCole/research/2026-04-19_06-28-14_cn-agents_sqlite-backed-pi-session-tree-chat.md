# Research: SQLite-backed Pi Session Tree Chat

## Answers

### 1. What exact session/tree structures and entry types does Pi persist in its append-only session model, and which of those are required to preserve v1 conversation fidelity when loading a session back into Pi from SQLite?

- Pi stores sessions as JSONL with a `session` header first, then append-only entries. The docs describe v1 as linear, v2 as tree-shaped via `id`/`parentId`, and v3 as the `hookMessage -> custom` rename. The header fields are `type`, optional `version`, `id`, `timestamp`, `cwd`, and optional `parentSession`. Loader code rejects files whose first valid record is not a `session` header with a string `id`. (`context/pi-mono/packages/coding-agent/docs/session.md:3-4`, `context/pi-mono/packages/coding-agent/docs/session.md:19-25`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:29-35`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:210-268`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:433-457`)

- Every non-header entry has `id`, `parentId`, and `timestamp`, and Pi currently persists these entry kinds: `message`, `thinking_level_change`, `model_change`, `compaction`, `branch_summary`, `custom`, `custom_message`, `label`, and `session_info`. `compaction` carries `summary`, `firstKeptEntryId`, `tokensBefore`, optional `details`, and optional `fromHook`; `branch_summary` carries `fromId`, `summary`, optional `details`, and optional `fromHook`; `label` carries `targetId`; `session_info` carries `name`. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:43-145`)

- `SessionMessageEntry.message` is typed as `AgentMessage`. The public session-file docs list `UserMessage`, `AssistantMessage`, `ToolResultMessage`, `BashExecutionMessage`, `CustomMessage`, `BranchSummaryMessage`, and `CompactionSummaryMessage` in that union, with JSON examples for `user`, `assistant`, and `toolResult`. In current persistence code, `message_end` writes `role === "custom"` as `custom_message`, and writes `user`, `assistant`, and `toolResult` as `message` entries. (`context/pi-mono/packages/coding-agent/docs/session.md:71-169`, `context/pi-mono/packages/coding-agent/docs/session.md:200-207`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:513-530`)

- Pi does not persist a separate active-head field. `_buildIndex()` reconstructs `leafId` by walking file order and assigning the last non-header entry as the active leaf. `buildSessionContext()` then walks `leaf -> parentId -> ... -> root` and only interprets `message`, `custom_message`, `branch_summary`, `compaction`, `thinking_level_change`, and `model_change` to rebuild runtime state. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:310-416`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:749-820`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1029-1045`)

- Tree navigation uses entry type to decide the new leaf. Selecting a user message or `custom_message` moves the leaf to that entry's parent and returns editor text; selecting a non-user entry moves the leaf to the selected node. Summarized navigation appends a `branch_summary` entry; unsummarized navigation only calls `branch()` or `resetLeaf()`, which are in-memory leaf moves. (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:2757-2816`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1120-1156`)

- Labels and session names are persisted separately from the LLM context path. `custom` entries are also explicitly documented as extension-state persistence that “does NOT participate in LLM context.” Label behavior is resolved into tree nodes and excluded from `buildSessionContext()` in both code and tests. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:87-100`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:905-929`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1001-1021`, `context/pi-mono/packages/coding-agent/test/session-manager/labels.test.ts:182-191`)

- The current restore path in `sdk.ts` consumes the `buildSessionContext()` result, restores model/thinking from it, and seeds `agent.state.messages` from it. That means the entries Pi actually reads during restore are the active-path entries that contribute to `buildSessionContext()`, plus a valid header that lets the file load at all. (`context/pi-mono/packages/coding-agent/src/core/sdk.ts:189-205`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:228-241`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:330-341`)

- Pi also has a built-in “current branch only” export path: `exportToJsonl()` writes a fresh header, takes `sessionManager.getBranch()`, and re-chains `parentId` linearly. That is current evidence that header + active branch path is sufficient for a linear “current conversation” export, while full tree fidelity requires preserving the original `id`/`parentId` graph and append order. (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:2994-3005`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:210-235`)

### 2. Given the desired semantics of distinct Pi sessions with shared ancestor history by reference, what data relationships are required to model threads, persisted session entries, fork points, active heads, and parent links in SQLite without flattening the tree into an invalid linear range model?

- The current app-side SQLite schema is linear. `chat_messages` belong to exactly one `thread_id` and are always read `ORDER BY created_at ASC`; `pipeline_artifacts` belong to one `pipeline_id` and carry an optional `session_path`; `openclaw_session_messages` stores a `parent_id` column but still reads rows by `session_key` and `timestamp ASC`. (`db/migrations/schema.sql:78-101`, `db/migrations/schema.sql:122-139`, `db/migrations/schema.sql:217-225`, `db/queries/chat_threads.sql:1-20`, `db/queries/chat_messages.sql:1-12`, `db/queries/pipeline_artifacts.sql:1-9`, `db/queries/openclaw_messages.sql:1-16`)

- Pi separates session identity from entry identity. The session header has its own session `id` and optional `parentSession`, while every persisted entry has its own `id` and `parentId`. The session selector then builds a cross-session tree from `parentSessionPath`, not from in-session entry edges. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:29-35`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:43-47`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:592-603`, `context/pi-mono/packages/coding-agent/src/modes/interactive/components/session-selector.ts:198-224`)

- Pi's active conversation state is defined by one nullable head/leaf. `buildSessionContext()` returns an empty context when `leafId === null`; otherwise it follows the parent chain from that leaf to root. The `SessionManager` class comment also describes the leaf pointer as the current position in the tree. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:310-347`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:656-675`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:964-970`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1127-1134`)

- The in-session graph is a true adjacency list, not a time slice. `getTree()` builds children from `parentId`, `getChildren()` returns direct descendants, and `branch()` changes only the leaf pointer so that the next append becomes a child of an earlier entry. Tests verify multiple siblings under one parent and nested branching. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:976-1107`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1115-1125`, `context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:167-243`, `context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:299-310`)

- Branch membership is ancestry-based, not “all rows between timestamps X and Y.” `buildSessionContext()` follows the selected leaf's parent chain only. The branch test expects `msg1/msg2/msg4-branch` and excludes sibling `msg3`, and `createBranchedSession(id5)` keeps `[id1,id2,id4,id5]` rather than a contiguous run of rows. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:341-416`, `context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:388-461`)

- Pi also persists non-parent cross references that are structurally meaningful: `branch_summary.fromId`, `compaction.firstKeptEntryId`, and `label.targetId`. Those relationships exist alongside `parentId`; they are not derivable from a flat thread/message ordering. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:66-84`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1001-1021`, `context/pi-mono/packages/coding-agent/docs/session.md:226-289`)

- Entry IDs are collision-checked only within the current session when generated, and `createBranchedSession()` writes the selected path entries unchanged into the new session file. That means the same entry IDs can reappear across multiple session files if history is shared by copy. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:200-207`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1165-1217`)

- The current `cn-agents` pipeline/session code does not encode Pi's required relationships. It derives sessions by scanning artifact rows for `session_path`, picks the requested path or the newest one, and builds active-session URLs from that path. (`server/services/pipelines/service.go:354-410`, `server/services/pipelines/service.go:437-525`, `server/services/pipelines/service.go:670-691`, `server/services/pipelines/session_logs.go:46-117`)

### 3. Which stable runtime hook points in the current Pi → TS worker → Temporal flow expose complete enough state to checkpoint the session into SQLite, and what exactly is available at `turn_end` versus fork/branch creation, artifact writes, and workflow completion?

- The Go workflow entry point is `RunStage()`. It constructs `PiActivityInput{pipeline_id, stage_id, prompt, cwd, base_dir, event_endpoint, thinking_level}`, executes the TS activity, then persists `SessionPath`/`Metadata` as a pipeline artifact and notifies the UI. (`pkg/agents/pipeline/types.go:17-34`, `pkg/agents/pipeline/run_stage.go:36-119`, `pkg/agents/pipeline/activities.go:116-143`)

- The TS worker creates or reopens Pi sessions and forwards every Pi event to Go. `RunPiAgent()` creates `SessionManager.create(input.cwd, input.base_dir)`, disables auto compaction, increments `metadata.turns` on `turn_end` and `metadata.tool_calls` on `tool_execution_end`, forwards `{type, data}` to the event endpoint, and returns `session_path` plus serialized metadata. `ContinuePiAgent()` requires `input.session_path`, opens that file, and returns the same shape. (`pkg/agents/temporal/workers/ts/activities.ts:33-93`, `pkg/agents/temporal/workers/ts/activities.ts:104-153`)

- The forwarded event payload shapes are defined in Pi's extension types. `TurnEndEvent` carries `turnIndex`, the final `message`, and `toolResults[]`. `AgentEndEvent` carries `messages[]`. `ToolExecutionEndEvent` carries `toolCallId`, `toolName`, `result`, and `isError`. (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:558-619`)

- Branch/fork lifecycle hooks are split into pre- and post-mutation events. `session_before_fork` only carries `entryId`. `session_before_tree` carries `targetId`, `oldLeafId`, `commonAncestorId`, `entriesToSummarize`, summary options, and optional label overrides. `session_tree` carries `newLeafId`, `oldLeafId`, and optional `summaryEntry`/`fromExtension` after the leaf has already been updated. (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:445-515`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:2695-2816`, `context/pi-mono/packages/coding-agent/src/core/agent-session-runtime.ts:179-246`)

- Pi's durable append boundary is `message_end`, not `turn_end`. The `message_end` handler appends `custom_message` for `role === "custom"` and appends `message` for `user`/`assistant`/`toolResult`. `turn_end` is emitted later as a completed-turn event for extensions/listeners. (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:513-530`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:602-626`)

- There is no dedicated Pi “artifact written” event in the current flow. The closest runtime payloads are tool lifecycle events such as `tool_execution_end`, while the durable pipeline artifact write occurs only after the TS activity returns and Go calls `PersistArtifact()`. (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:614-619`, `pkg/agents/pipeline/run_stage.go:92-119`, `pkg/agents/pipeline/activities.go:116-143`)

- Workflow-completion state is split across Pi and Go. Pi exposes `agent_end.messages[]`; the TS activity returns `PiActivityOutput{artifact_path, session_path, metadata}`; Go then persists `PersistArtifactInput{FilePath, SessionPath, ExecutionMetadata}` and returns `StageOutput{ArtifactPath, SessionPath, ExecutionMetadata}`. (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:558-560`, `pkg/agents/temporal/workers/ts/activities.ts:89-93`, `pkg/agents/temporal/workers/ts/activities.ts:145-153`, `pkg/agents/pipeline/run_stage.go:98-119`, `pkg/agents/pipeline/types.go:29-34`, `pkg/agents/pipeline/types.go:58-65`)

- Browser streaming goes through the internal event endpoint and pipeline notifier. `/internal/events` binds the forwarded `AgentEvent`, `PipelineNotifier.NotifyEvent()` fans it out to pipeline subscribers, and `streamPipelineEvents()` patches `artifact-events-pane` with the accumulated event list. (`server/services/pipelines/handler.go:207-237`, `server/services/pipelines/handler.go:549-563`, `server/services/pipelines/notifier.go:8-12`, `server/services/pipelines/notifier.go:95-109`)

### 4. Where is continuation currently coupled to review-stage workflows and filesystem `session_path` persistence, and what current assumptions in `pkg/agents` would prevent resuming the latest head or forking from an arbitrary prior message by starting a brand-new Temporal workflow from SQLite-backed state?

- QRSPI continuation is review-stage-centric. Each agent stage has a paired review stage in the registered workflow type, and interactive agent stages advance into a review stage instead of directly starting the next agent stage. (`pkg/agents/workflows/qrspi/workflow.go:34-48`, `pkg/agents/workflows/qrspi/workflow.go:63-65`, `pkg/agents/workflows/qrspi/workflow.go:82-84`, `pkg/agents/workflows/qrspi/workflow.go:101-103`, `pkg/agents/workflows/qrspi/workflow.go:120-122`, `pkg/agents/workflows/qrspi/workflow.go:138-140`)

- The active “continue” path is inside review workflows. `runReviewWorkflow()` and `runTerminalReviewWorkflow()` wait on `continue/approve/reject` signals, load the reviewed stage's previous artifact, copy `previousArtifact.SessionPath` into `PiActivityInput.SessionPath`, and call `ContinuePiAgent`. (`pkg/agents/workflows/qrspi/workflow.go:169-251`)

- Temporal signaling is modeled around review decisions, not session-tree state. The only signal names are `continue`, `approve`, and `reject`; `ContinuePipeline()` derives a deterministic workflow ID for the current stage and signals that workflow. (`pkg/agents/temporal/manager.go:17-36`, `pkg/agents/temporal/manager.go:101-103`, `pkg/agents/temporal/manager.go:124-126`, `server/services/pipelines/service.go:207-220`)

- The HTTP “resume session” path validates both `message` and `session_path`, but `ResumeSessionWithPrompt()` ignores the chosen session after validation and just calls `ContinuePipeline()`. (`server/services/pipelines/handler.go:279-295`, `server/services/pipelines/service.go:222-243`)

- Across the Go↔TS boundary, the only continuation cursor is a string `session_path`. `PiActivityInput` has no field for branch ID, leaf ID, message ID, or fork point, and `PersistArtifactInput` stores only a `SessionPath` string alongside the artifact path. (`pkg/agents/pipeline/types.go:17-27`, `pkg/agents/pipeline/types.go:58-65`)

- `ContinuePiAgent()` enforces that file-path assumption directly: it throws if `input.session_path` is empty, then calls `SessionManager.open(input.session_path)` and immediately prompts. (`pkg/agents/temporal/workers/ts/activities.ts:104-153`)

- The current stage page, session picker, and session-log viewer are also keyed by path strings. The UI posts hidden `session_path` fields, `ResolveActiveSession()` returns the requested path or the newest artifact-derived path, `BuildArtifactPageArgs()` builds `ActiveSessionPath` and stream URLs from that string, and session-log resolution prefers artifact `session_path` then falls back to scanning `.jsonl` files on disk. (`server/services/pipelines/templates.templ:378-390`, `server/services/pipelines/templates.templ:444-452`, `server/services/pipelines/service.go:392-410`, `server/services/pipelines/service.go:455-525`, `server/services/pipelines/service.go:670-691`, `server/services/pipelines/session_logs.go:46-117`)

- Starting new Temporal workflows is stage-driven, not session-driven. `StartNextWorkflow()` accepts `StageInput`, derives a deterministic stage workflow ID, and never reads session-tree/head state. (`pkg/agents/pipeline/activities.go:73-107`, `pkg/agents/temporal/manager.go:124-126`)

### 5. What existing Datastar/templ surfaces from the current chat and pipeline pages can be reused for a v1 UI with a threads-only sidebar, streamed chat transcript, and a workflow-defined working-directory artifact viewer, and which pieces remain host-app-specific rather than reusable SDK/host integration points?

Their argument types are narrow: the sidebar only needs `Threads` plus `CurrentThreadID`, and the message components only need message IDs, roles, and rendered content. (`server/services/chat/templates.templ:3-84`, `server/services/chat/args.go:17-56`)

(`server/services/chat/handler.go:165-348`)

- The current pipeline stage page already composes a left pane of artifact navigator + session pane + session log + event panel and a right pane of rendered markdown. `ArtifactPageArgs` and `ArtifactSessionState` are the data contracts that feed those panes. - The pipeline pages bind the panes to workflow types, artifact discovery, `session_path`-based selection, review actions, event streaming, and URL generation. (`server/services/pipelines/service.go:455-525`, `server/services/pipelines/service.go:632-691`, `server/services/pipelines/handler.go:279-333`, `server/services/pipelines/handler.go:351-357`, `server/services/pipelines/handler.go:478-563`)

### 6. If v1 restore only needs the conversation/tree state that Pi normally writes to JSONL, what non-conversation session data can safely be deferred without breaking latest-resume and fork-from-message behavior, and which deferred features would materially affect the fidelity of a SQLite-backed restore?

- The current restore path is driven by `buildSessionContext()`, then `sdk.ts` restores model, thinking, and `agent.state.messages` from that result. `buildSessionContext()` only interprets `message`, `custom_message`, `branch_summary`, `compaction`, `thinking_level_change`, and `model_change`. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:310-416`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:189-205`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:228-241`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:330-341`)

- Session titles can be deferred for core restore mechanics. `session_info` is appended/read by `appendSessionInfo()`/`getSessionName()`, but it is not used by `buildSessionContext()` or by runtime fork logic. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:905-929`, `context/pi-mono/packages/coding-agent/src/core/agent-session-runtime.ts:179-246`)

- Labels can also be deferred for core restore mechanics. Labels are stored as separate `label` entries, resolved into tree-node display state, and explicitly excluded from `buildSessionContext()` in tests. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1001-1021`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1075-1079`, `context/pi-mono/packages/coding-agent/test/session-manager/labels.test.ts:182-191`)

- `custom` extension entries are deferrable only if extension-state fidelity is out of scope. The type comment says they persist extension state across reloads and “does NOT participate in LLM context.” (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:87-100`)

- Model changes, thinking-level changes, compaction, branch summaries, and custom messages are not safely deferrable if restore fidelity matters. `buildSessionContext()` reads them, and `sdk.ts` uses the resulting model/thinking/message state during restore. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:354-407`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:189-205`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:228-241`, `context/pi-mono/packages/coding-agent/src/core/sdk.ts:330-341`)

- `cwd` is also not safely deferrable for workspace-correct resume. `SessionManager.open()` restores `cwd` from the session header, and the runtime tests assert that switching sessions updates both `process.cwd()` and the runtime's session cwd. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1274-1281`, `context/pi-mono/packages/coding-agent/test/suite/agent-session-runtime.test.ts:236-304`)

- `parentSession` is not used by `open()` or `fork()` mechanics, but it is consumed when building the threaded session selector tree. Deferring it does not change `buildSessionContext()`, but it does change cross-session lineage display. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:29-35`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:592-603`, `context/pi-mono/packages/coding-agent/src/modes/interactive/components/session-selector.ts:198-224`)

- Session-list fields such as `messageCount`, `firstMessage`, `allMessagesText`, and `modified` are derived metadata rather than restore inputs. Pi computes them from entries/header/stat data and sorts session lists by `modified` descending. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:506-607`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:1359-1363`, `context/pi-mono/packages/coding-agent/src/modes/interactive/components/session-selector.ts:430-441`)

### 7. What automated tests and runtime verification scenarios must exist to prove that completed turns are durably checkpointed, shared-history forks create correct new threads, latest-resume always spins up a new Temporal workflow from SQLite, and the SSE-driven chat/artifact UI stays consistent across resume and fork actions?

- The current repo already has automated checks for Pi's core session-tree invariants. Tree tests verify parentId chains, sibling branches, branch summaries, branch-only context reconstruction, and `createBranchedSession()` extracting the selected ancestry path. (`context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:10-29`, `context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:167-243`, `context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:299-339`, `context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:388-461`)

- Durable file behaviors are also covered at the Pi session-file layer. Tests verify that files without a valid header load as empty, corrupted/empty files are rewritten with a valid header, forked sessions without an assistant delay file creation until the first assistant response, and forked paths that already include an assistant write immediately. (`context/pi-mono/packages/coding-agent/test/session-manager/file-operations.test.ts:19-39`, `context/pi-mono/packages/coding-agent/test/session-manager/file-operations.test.ts:141-206`, `context/pi-mono/packages/coding-agent/test/session-manager/tree-traversal.test.ts:476-529`)

- Fork behavior is covered in runtime tests as well. ## Surprises
- The active leaf/head is implicit in append order, not persisted as a separate field. `_buildIndex()` simply makes the last non-header entry the leaf on load. (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:749-757`)
- `ResumeSessionWithPrompt()` validates a user-selected `session_path`, but the implementation immediately falls through to `ContinuePipeline()` and never passes that session path into a fresh Temporal workflow start. (`server/services/pipelines/service.go:222-243`)
- `createBranchedSession()` reuses the selected path entries unchanged, so entry IDs are only session-local at creation time and can recur across different session files.
