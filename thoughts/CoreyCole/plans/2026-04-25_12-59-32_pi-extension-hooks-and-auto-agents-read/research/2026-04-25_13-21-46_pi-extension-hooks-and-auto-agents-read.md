# Research: Pi extension hooks and automatic AGENTS.md reads

## Brainstorm Summary
- The question doc defines the desired outcome as two separate Pi extensions in the personal dotfiles config: one for configurable pre/post tool hooks and one for automatic `AGENTS.md` loading tied to `read` tool usage (`thoughts/CoreyCole/plans/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read/questions/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read.md:17-22`).
- The question doc also fixes several constraints for the auto-read behavior: trigger only on `read`, walk ancestor `AGENTS.md` files, emit visible reads in session history, dedupe by exact absolute path, and re-read when file content changes (`thoughts/CoreyCole/plans/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read/questions/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read.md:21-22`).
- Research for this change set is supposed to use `context/pi-mono` as ground truth and load the Pi skill rather than relying on memory (`thoughts/CoreyCole/plans/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read/questions/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read.md:19-20`).
- The open design tension deferred by the question doc is whether the AGENTS behavior belongs in `tool_call`/`tool_result`, a built-in `read` override, or a deeper resource/session integration point (`thoughts/CoreyCole/plans/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read/questions/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read.md:22-22`).

## Research Question
What Pi extension/runtime primitives, context-loading paths, session-state mechanisms, and visible tool-history paths already exist for implementing Claude-Code-style tool hooks and a `read`-triggered automatic `AGENTS.md` read flow in this dotfiles Pi config (`thoughts/CoreyCole/plans/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read/questions/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read.md:27-34`)?

## Summary
Pi already exposes generic pre/post tool interception through `tool_call` and `tool_result`, plus non-mutating execution lifecycle notifications through `tool_execution_start` / `tool_execution_update` / `tool_execution_end` (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:373-416`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:655-676`, `context/pi-mono/packages/coding-agent/src/core/extensions/runner.ts:710-777`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:680-700`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:772-991`). Pi’s current `AGENTS.md` loading is startup-time prompt assembly: `loadProjectContextFiles()` loads one context file per directory, preferring `AGENTS.md` over `CLAUDE.md`, with the global agent file first and ancestor files ordered outermost-to-innermost before `buildSystemPrompt()` appends them as `# Project Context` (`context/pi-mono/packages/coding-agent/src/core/resource-loader.ts:58-113`, `context/pi-mono/packages/coding-agent/src/core/resource-loader.ts:455-459`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:909-926`, `context/pi-mono/packages/coding-agent/src/core/system-prompt.ts:28-171`). The built-in `read` tool is not special in the extension layer, so read-specific behavior can only be attached via generic tool hooks or by replacing the built-in `read` registration (`context/pi-mono/packages/coding-agent/src/core/tools/read.ts:121-267`, `context/pi-mono/packages/coding-agent/examples/extensions/built-in-tool-renderer.ts:14-18`, `context/pi-mono/packages/coding-agent/examples/extensions/built-in-tool-renderer.ts:37-94`). Session-scoped extension state can be persisted either as `custom` entries that do not enter LLM context or as `custom_message` entries that do, and local dotfiles extensions already use both patterns (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:91-135`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:897-958`, `.pi-config/extensions/review.ts:42-77`, `.pi-config/extensions/review.ts:812-812`, `.pi-config/extensions/review.ts:1055-1138`).

## Detailed Findings

### 1. Existing Pi primitives for pre-tool and post-tool interception
`AgentSession` installs tool hooks directly onto the underlying agent. `beforeToolCall` emits a `tool_call` event and `afterToolCall` emits a `tool_result` event through the current `ExtensionRunner`, so extension reload swaps runners without reinstalling the hooks (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:373-416`). Before `tool_call` runs, `AgentSession` waits for the agent event queue to drain (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:380-386`).

`tool_call` is Pi’s pre-execution interception surface. The type comment says `event.input` is mutable, earlier handlers’ mutations are visible to later handlers, and no re-validation happens after mutation (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:804-811`). The only structured return value on `tool_call` is blocking metadata: `block?: boolean` and `reason?: string` (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:974-977`). `ExtensionRunner.emitToolCall()` passes the same event through handlers in extension order and returns early on the first blocking result (`context/pi-mono/packages/coding-agent/src/core/extensions/runner.ts:760-777`).

`tool_result` is Pi’s post-execution mutation surface. `ToolResultEventResult` can patch `content`, `details`, and `isError` (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:988-991`). `ExtensionRunner.emitToolResult()` copies the incoming event once, then applies those returned fields cumulatively across handlers (`context/pi-mono/packages/coding-agent/src/core/extensions/runner.ts:710-757`).

Pi also emits `tool_execution_start`, `tool_execution_update`, and `tool_execution_end`, but those are forwarded as normal extension events rather than dedicated mutation hooks (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:655-676`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:680-700`). They provide observability around actual execution, not a block/patch return channel.

Regression tests lock the hook behavior: one test blocks tool execution via `tool_call`, and another modifies tool results via `tool_result` (`context/pi-mono/packages/coding-agent/test/suite/agent-session-model-extension.test.ts:96-175`).

### 2. Current Pi mechanism for loading `AGENTS.md` and how it differs from per-`read` loading
Pi’s current context-file loading happens in `loadContextFileFromDir()` and `loadProjectContextFiles()`, not in the `read` tool. For any single directory, `loadContextFileFromDir()` checks `AGENTS.md` before `CLAUDE.md` and returns the first match only (`context/pi-mono/packages/coding-agent/src/core/resource-loader.ts:58-72`). `loadProjectContextFiles()` then adds the global agent-dir context first, walks from `cwd` upward, unshifts ancestor hits so final order is outermost ancestor to current directory, dedupes by exact path string, and appends the ancestor list after the global file (`context/pi-mono/packages/coding-agent/src/core/resource-loader.ts:76-113`).

`DefaultResourceLoader.reload()` stores those files in `this.agentsFiles` unless `noContextFiles` is true (`context/pi-mono/packages/coding-agent/src/core/resource-loader.ts:455-459`). `AgentSession._rebuildSystemPrompt()` copies `resourceLoader.getAgentsFiles().agentsFiles` into `BuildSystemPromptOptions.contextFiles` (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:909-926`). `buildSystemPrompt()` then appends a `# Project Context` section and renders each context file as `## path` followed by its raw content (`context/pi-mono/packages/coding-agent/src/core/system-prompt.ts:28-77`, `context/pi-mono/packages/coding-agent/src/core/system-prompt.ts:149-169`).

This differs from the requested behavior in several concrete ways. Current Pi context loading is startup/reload prompt assembly, not lazy loading on `read` (`context/pi-mono/packages/coding-agent/src/core/resource-loader.ts:455-459`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:909-926`). Current Pi chooses one context file per directory and may load `CLAUDE.md`; the request specifically talks about per-`read` ancestor `AGENTS.md` handling (`context/pi-mono/packages/coding-agent/src/core/resource-loader.ts:58-72`, `thoughts/CoreyCole/plans/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read/questions/2026-04-25_12-59-32_pi-extension-hooks-and-auto-agents-read.md:21-22`). Current Pi also does not create visible read-tool history entries when context files are loaded; it only injects them into the system prompt and shows a startup context summary in the UI (`context/pi-mono/packages/coding-agent/src/core/system-prompt.ts:60-66`, `context/pi-mono/packages/coding-agent/test/interactive-mode-status.test.ts:788-825`).

### 3. Existing implementation options for attaching behavior to `read`
The built-in `read` tool is defined like any other built-in tool. It exposes `{ path, offset?, limit? }`, resolves the file path relative to the tool cwd, returns either image content or truncated text, and defines `renderCall()` / `renderResult()` for UI rendering (`context/pi-mono/packages/coding-agent/src/core/tools/read.ts:17-31`, `context/pi-mono/packages/coding-agent/src/core/tools/read.ts:121-267`). Because `ReadToolCallEvent` and `ReadToolResultEvent` are just typed cases inside the generic tool event unions, extensions can target `read` by filtering on `toolName === "read"` inside `tool_call`, `tool_result`, or `tool_execution_*` handlers (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:772-775`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:836-838`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:680-700`).

The first option is generic event handling. `tool_call` can mutate read arguments or block the call, while `tool_result` can rewrite the returned content/details/error status (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:804-811`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:974-991`, `context/pi-mono/packages/coding-agent/src/core/extensions/runner.ts:710-777`). The tradeoff is that the underlying visible tool row is still the original read execution unless the extension separately injects custom session messages.

The second option is built-in tool replacement. The built-in renderer example states that registering a tool with the same name as a built-in replaces it entirely, and then demonstrates re-registering `read` while delegating execution to `createReadTool(cwd)` (`context/pi-mono/packages/coding-agent/examples/extensions/built-in-tool-renderer.ts:14-18`, `context/pi-mono/packages/coding-agent/examples/extensions/built-in-tool-renderer.ts:37-94`). That replacement surface gives full control over read execution and rendering, but it is no longer just a post-hoc event hook.

The third option is execution-lifecycle observation. The agent loop emits `tool_execution_start` before execution, optional `tool_execution_update` during streaming updates, and `tool_execution_end` after execution (`context/pi-mono/packages/agent/src/agent-loop.ts:360-403`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:655-676`). Those events are useful for coordination or UI side effects, but not for argument/result mutation.

### 4. Where session-scoped state can live for dedupe and content-hash tracking
Pi exposes `appendEntry()` on the extension API, and `AgentSession` wires it to `sessionManager.appendCustomEntry()` (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:1177-1177`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:2166-2168`). `CustomEntry` is explicitly documented as persistent extension state that does not participate in LLM context (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:91-101`). `appendCustomEntry()` stores `type: "custom"`, `customType`, `data`, `id`, `parentId`, and `timestamp` in the session tree (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:897-905`).

Pi also supports `CustomMessageEntry` for state that must participate in LLM context. The comments distinguish it from `CustomEntry`: it does participate in context, and `appendCustomMessageEntry()` stores `type: "custom_message"`, `customType`, `content`, `display`, `details`, `id`, `parentId`, and `timestamp` (`context/pi-mono/packages/coding-agent/src/core/session-manager.ts:121-135`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:945-958`). When messages are converted for the LLM, custom messages are mapped to user-role messages; plain custom entries are ignored (`context/pi-mono/packages/coding-agent/src/core/messages.ts:123-131`, `context/pi-mono/packages/coding-agent/src/core/messages.ts:142-167`).

The local `review.ts` extension is the clearest in-repo persistence example. It reconstructs state by scanning branch entries for `type === "custom"` and `customType === "review-session"` (`.pi-config/extensions/review.ts:42-77`). It persists state transitions with `pi.appendEntry(REVIEW_STATE_TYPE, { active: true, originId: ... })` and clears them with `active: false` entries (`.pi-config/extensions/review.ts:812-812`, `.pi-config/extensions/review.ts:1055-1138`). That demonstrates that per-session extension state can be recovered from the session branch itself rather than only from module-level memory.

### 5. Code paths and formats for a visible `AGENTS.md` auto-read in session history
Pi’s visible tool history is driven by the agent loop and the session persistence path. The agent loop emits `tool_execution_start`, finalizes the tool call, emits `tool_execution_end`, then creates a `role: "toolResult"` message with `toolCallId`, `toolName`, `content`, `details`, `isError`, and `timestamp`, and finally emits `message_start` / `message_end` for that tool result (`context/pi-mono/packages/agent/src/agent-loop.ts:360-403`, `context/pi-mono/packages/agent/src/agent-loop.ts:440-482`, `context/pi-mono/packages/agent/src/agent-loop.ts:668-682`).

`AgentSession` persists those `toolResult` messages when it receives `message_end`; user, assistant, and tool-result messages are stored through `sessionManager.appendMessage()` (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:525-541`). That means visible tool history in the session naturally comes from real tool executions that produce tool-result messages, not from silent prompt mutation.

The interactive tool UI is rendered by `ToolExecutionComponent`, which binds tool name, args, execution state, optional partial results, and final results to built-in or extension renderers (`context/pi-mono/packages/coding-agent/src/modes/interactive/components/tool-execution.ts:1-183`). The built-in `read` tool already provides `renderCall()` and `renderResult()` (`context/pi-mono/packages/coding-agent/src/core/tools/read.ts:258-266`).

For a visible AGENTS auto-read that should look like tool history rather than hidden prompt mutation, the existing Pi code paths point to producing actual tool/session-visible output: either by causing a real read-tool execution path to occur or by inserting a `custom_message` that is intentionally displayed (`context/pi-mono/packages/agent/src/agent-loop.ts:668-682`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:525-541`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:121-135`, `context/pi-mono/packages/coding-agent/src/core/session-manager.ts:945-958`).

### 6. Mapping Claude Code hook semantics onto Pi’s settings and extension model
Official Claude Code hook docs describe hooks as configuration-driven lifecycle handlers with session-level, turn-level, and tool-level events. The hooks reference says the once-per-tool events are `PreToolUse` and `PostToolUse`, while other lifecycle events include `SessionStart`, `SessionEnd`, `UserPromptSubmit`, `Stop`, `StopFailure`, and `InstructionsLoaded` (https://docs.anthropic.com/en/docs/claude-code/hooks). The same page says `PreToolUse` can block a tool call and that hooks are configured in JSON settings files rather than in extension code (https://docs.anthropic.com/en/docs/claude-code/hooks).

Pi has direct analogs for the tool-level lifecycle but not the same declarative configuration shape. The closest Pi equivalents are `tool_call` for pre-tool interception, `tool_result` for post-tool result mutation, and `session_start`, `agent_end`, `input`, and `before_agent_start` for broader lifecycle interception (`context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:680-700`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:804-811`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:974-991`, `context/pi-mono/packages/coding-agent/src/core/extensions/types.ts:1100-1112`). Pi settings currently load extension entry points from `settings.json`; in this repo the tracked settings explicitly load only `+extensions/answer.ts` and `+extensions/execute-command.ts` (`.pi-config/agent/settings.json:15-17`).

So the directly mappable part is the lifecycle surface: Claude `PreToolUse` maps most closely to Pi `tool_call`, and Claude post-tool behavior maps most closely to Pi `tool_result` plus `tool_execution_end` depending on whether mutation is needed (`context/pi-mono/packages/coding-agent/src/core/agent-session.ts:373-416`, `context/pi-mono/packages/coding-agent/src/core/agent-session.ts:655-676`). The non-matching part is the configuration mechanism: Claude’s docs describe settings-defined matcher groups and handlers, while Pi exposes code-level extension events and extension loading through settings rather than a built-in hook DSL (https://docs.anthropic.com/en/docs/claude-code/hooks, `.pi-config/agent/settings.json:15-17`).

### 7. Tests, examples, and nearby patterns that should guide implementation and verification
For hook behavior, the most directly relevant Pi tests are the extension hook regression tests: `agent-session-model-extension.test.ts` proves `tool_call` can block execution and `tool_result` can modify results (`context/pi-mono/packages/coding-agent/test/suite/agent-session-model-extension.test.ts:96-175`). `resource-loader.test.ts` verifies AGENTS discovery and `noContextFiles` behavior (`context/pi-mono/packages/coding-agent/test/resource-loader.test.ts:269-287`). `interactive-mode-status.test.ts` verifies that the UI shows the global context file before the project file and shortens paths relative to cwd in the compact display (`context/pi-mono/packages/coding-agent/test/interactive-mode-status.test.ts:788-825`).

For examples in `context/pi-mono`, `permission-gate.ts` is the clearest `tool_call` blocking example (`context/pi-mono/packages/coding-agent/examples/extensions/permission-gate.ts:13-28`). `built-in-tool-renderer.ts` is the clearest built-in-tool replacement example because it re-registers `read` by name and delegates to the original implementation (`context/pi-mono/packages/coding-agent/examples/extensions/built-in-tool-renderer.ts:14-18`, `context/pi-mono/packages/coding-agent/examples/extensions/built-in-tool-renderer.ts:37-94`).

## Open Questions
